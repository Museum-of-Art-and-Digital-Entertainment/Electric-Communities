What These Catalogs Are About

"Sometimes, it's more important to be reminded than informed."
-Warren Teitelman, ICSE, 1983 or 1984

If you spend time analyzing bugs, you'll find that a certain significant percentage of them are repeats: bugs that have been made before. A couple of classic examples:
  • Off-by-one errors.
  • Forgetting to check for error returns from write(). (Sometimes that's not worth doing, but it should be a conscious decision, not an oversight. I once heard of a backup product that failed to check whether a write failed because the tape filled up. Guess how the bug was discovered?)

These particular catalogs are for programmers or for testers poking around in the code. In keeping with the quote above, their purpose is less to teach than to jog you into remembering something you know, but happened to overlook. To use them:
  1. Pick a catalog. For example, if you're writing E code, you'll use the E catalog. If your code uses a particular ingredient, you'll use that ingredient's catalog.
  2. Scan through the entries, deciding which might apply to your code. For example, if you use E exceptions, you'd use the "E exceptions" entry.
  3. Each entry contains a series of hints. For example, the "E exceptions" entry contains:

    Ethrow is a message send.
    Ecatch creates a closure.
    …

    You can use those hints as a checklist. Use them before or after you write the code, as you prefer. A hint might also suggest a test case. (Sometimes it's easier or more reliable to check that the code works by reading it, sometimes by executing it.) The first time you read a hint, it might not mean much, but there's explanatory text below each one. There may also be a hyperlink to an example.
