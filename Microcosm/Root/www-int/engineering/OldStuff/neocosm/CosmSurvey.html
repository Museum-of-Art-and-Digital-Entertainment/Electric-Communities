As part of figuring out the best way to move forward we need to understand what we have. The application can be divided into subsystems, and we need to survey them and the code that we have. Doing this will help get a good comprehensive plan of the work ahead.

If you are documenting a subsystem please use the Subsystem Document Template.

Once we have sorted out who will be working on which subsystems we will need to define requirements, issues, and decide on the future of legacy code, identifying carefully what is the functionality that we need, as well as possible alternatives.

E and its replacement

    E's Semantics [MarkM+: Arturo]

    • The RunQ.
    • RtTethers including EUniChannels, EStone and EProxy.

      • ELib implements the core runtime providing the basic E semantics.  This is checked in and is being maintained under our chose pure Java IDE.

    Core E language support [Chip*: Robj]

    • All the core language eclasses EInteger, EBoolean, etc.

      • We do not need these so much given the new E runtime; they will mostly go away and not be missed.

    Comm System [Bill*: Robj, Bob]

    • Point-to-point E comm system:
    • Encrypted links.
    • Unforgeable network object references with handoff logic.
    • SturdyRefs and the Registrar.
    • Message encode/decode.

      • The design for this is well underway.  It looks now like we will be reusing almost all the logic and good portions of the code of the current comm system, with the exception of the encoding system; we'll use standard Java serialization instead, which will reduce the amount of code we have to maintain.

UI Subsystems

    Display

      Renderer Subsystem [Mike: Claire, Tony]

      • The current plan is to use the existing AnyChannel renderer. See the Renderer Analysis page for a list of the issues involved in maintaining the AnyChannel renderer and porting to a new one.

      Dynamics Engine [Tony, Harry, Mike): Claire, Tony]

      • This is in pure Java but has always been a major source of deadlock bugs, and should be redesigned to greatly reduce their incidence. One big problem is the Dynamo3D code which is possibly not shippable; redoing this will be a sizable piece of work.

      ECFonts (the in-world text/balloon generator) [Walendo:?]

      • The code could use some cleanup and a review, but it's been stable for a long time. If we will want the same in-world text styles, there's probably no need to change the native code. The Java wrappers would need to change to reflect any new calling paradigms or renderer interfaces, should we change them.

        • Note that ECFonts just makes the graphics, it doesn't put them on the screen.

    Sound [Ronin: Claire]

      DirectSound Java support.

      VoxChat.

      • It looks plausible that we could just use the NetMeeting SDK instead of TNT's VoxChat (which is no longer supported at all, and thus basically unshippable).  The problem would then be, as it currently is, using chat while listening to sounds; we don't know if DirectSound coexists any better with NetMeeting's call API than it did with VoxChat (which was "not").

    UI Support [Claire: Tony, Arturo, Karl, Robj]

      IFC

      • The likeliest contender here is either Swing or JFC, as IFC is unsupported and WFC would require using all Microsoft tools.

      StoneLook

      • It is undecided whether we even continue to have a custom StoneLook-esque interface.  It would certainly expedite matters to be able to use an off-the-shelf GUI builder....

Pluribus and its replacement

    Introduction to containment issues [Robj* : Scott, Arturo]

    • The document itself lays out most of the issues here.  Generally we need something smaller, simpler, and easier to verify as correct.  The goal is to avoid some of the hard problems we don't completely solve in the current code, while leveraging as much of the current logic as makes sense.  The code itself is likely to need a complete rewrite to get to pure Java.
