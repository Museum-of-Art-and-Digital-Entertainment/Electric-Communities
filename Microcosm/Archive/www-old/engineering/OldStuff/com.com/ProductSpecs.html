ECID Specification

1.0 Introduction

ECID (Electric Communities Identity) is an identity system providing persistent, addressable, secure distributed identities for Internet processes representing people, places and things. ECIDs are globally unique without requiring the use of a central identity server. They use a collection of Process Location Servers to resolve connections from dynamic IP addresses. Process Location Servers can be public or private and can reside anywhere on the Internet. ECID relies on secure, encrypted communications between processes. Because it is decentralized, it can be added to any existing application.

With secure communications, all other applications can be associative optional.

ECID is used by end-users to manage only what you send out, not what you are watching.

ECID is used by organizations to manage and communicate with lists of users who have visited a WWW site, registered for a WWW-site or application.

ECID hands out capabilities.

ComCom Specification

1.0 Introduction

ComCom is a low cost, always up tool to help users coordinate activities between multiple people and places on the net in a secure fashion.

ComCom provides a simple, low cost/always up interface for getting basic information about what is going on in all the Internet contexts you frequent, including all of EC's venues. Offers opportunities for advertising and promotion of all virtual community and chat solutions. Provides value-added for EC technologies. Spying can have low-cost lobby-like effects: I can watch until my favorite partner comes to the Quake server and then I can launch the game to play together. We aggregate global user statistics -- it serves a mega @where function to know what is going on in all sites. We can know how many players are playing Quake, etc. The Spy Windows can come pre-loaded with links to WWW pages to promote our products or the products of our partners. It shows you how to download things you may never have considered using.

Goals and Themes

  • Abstract out the embedded features of the Microcosm Address Book, Teleport List, Notifications and The Palace's Little Buddy with a general solution that works across many application types.
  • Aggregate population quickly by providing high-level, immediate value to end-users, WWW sites, application developers and community builders.
  • Add features for managing security, people, places and groups.
  • Broadly leverage our expertise in virtual communities, chat and communication protocols.
  • Mine maximum, general value from our identity model.
  • Add multiple points of presence to the single point of presence of ICQ.
  • Down the road, this approach allows us to co-opt things like "Net Nanny" by setting preferences on identities. It allows us to CO-OPT HTTP by adding pseudonymity to what is an annonymous protocol now. It allows us to co-opt email by replacing domains with capabilities that sort mail. This also allows us to add reputation to identity on the 'Net.
  • Controls who and what I want to know about. It makes requests and hands out permissions.
2.0 Marketing

Could give WWW site owners the toolkit for free for ninety days so that they could get a taste of managing a chat or BBS. This way we get our button on their site for ninety days and they get free chat for ninety days. It is important to demonstrate the difference between ordered and disorderly sites.

  1. We tell WWW owners that chat is coming to their site for free. If they will put a ComCom button on their WWW site, we will share data with them and give them moderator functions. Otherwise, we essentially own their chat site. If the user who pushes this button has ComCom, they are instantly launched into the text-only ComCom dynachat session (text-only InstantPalace that downloads with our product. If the user does not have ComCom, they are sent to our page so they can download it. Once they download, they connect to the session. If the WWW owner wants more control or richness, they can pay us to turn on the graphics, get more wizard functions, get user data about their site from ComCom, control what the protocol releases to the Spy window, or even control what application is used for chat, etc.
  2. Or, we give chat away for free to these sites in return for them putting up a ComCom button, which gives them control of the chat site. This makes for a slower growth curve, as only sites interested would be chatted about.
  3. If Palace, then: when ComCom is loaded, people get Instant Palace for free. This helps out The Palace by making it nicer because people don't have to download the applet everytime they join a presentation. Using Palace technologies helps with development time and provides a smooth growth curve on Palace functions for WWW pages to turn into Internet Communities. We sell them up from the text chat to graphics chat to scripting and wizard and auditorium functions, always giving them means to address their population via the EC ID (e.g., that WWW site gets a group ID that collects the swiss IDs of people who have chatted there)
  4. At all points, we are collecting information about where people are and with whom. We become like Neilson and sell these data.
  5. We aggregate a population of users who are prime customers for our enterprise and consumer products, making marketing much easier.
  6. We aggregate a population of users who have value to others and get nice offers like ICQ got from AOL.
  7. We own the WWW WRT to identity and start to add tools that add value to identity, like client-side certificate managers, etc.
  8. Content can be pushed from the protocol to the ComCom window for advertising.
  9. EC owns a ComCom window for advertising and promotions.
  10. Game company WWW site visitors can start interacting about the game before they have the software via ComCom.

6.0 Functionality and Features

Features Requirements

  • Back-end designs that support partners must also aggregate ComCom users to EC.
  • Users cannot be in the same place at the same time with the same ID.
  • Adding new protocol types must be easy and publically published. Offer a standard scripting language/APIs/preferences that others can use to present their stuff through ComCom, including: Worlds Away, MPath, Quake, Yahoo Chat, Lotus Notes, NetMeeting, Microcosm, The Palace, Active Worlds, Worlds Chat, ICQ, IPage, IChat, People Link, AOL Instant Messager. We build these links for the first round of strategic populations with a goal to aggregate as quickly as possible.
  • ECID process migration issues must be addressed.
  • ComCom must be agnostic regarding the identity system used. ECIDs are only one type of identity that it can handle. This fosters its usefulness with existing legacy systems for enterprise usage. Must work with
  • ComCom must deal explicitly and intuitively with privacy issues at all levels. Users must not misunderstand what they are doing and revealing.
  • How do we make sure that with every back-end implementation choice that we are not cutting ourselves out of collecting data that may be valuable to sell?
  • Have to be able to distribute the domain so that corporations can run their own buddy lists. We always planned to have PLS’s that could be run by an IS person in a corporation. Has to be able to be shipped and partitioned.
  • Directory service needs to be identity neutral.
  • Have to deal with the scalability issues.
  • Has to run through firewalls – HTTP tunneling – has OnLive created a generic tunneling library to wrap the TCP-IP in? They are not doing this in Java.
  • No reboot to install.
