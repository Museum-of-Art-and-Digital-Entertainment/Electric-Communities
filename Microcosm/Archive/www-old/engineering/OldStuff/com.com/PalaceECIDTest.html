Capabilities

  • Support for locating a potential buddy and initializing the capability request. (Ess.)
  • Support for requesting a ‘where are you/esp?’ capability from another ECID. (Ess.)
  • Support for granting a ‘where are you/esp?’ capability request to another ECID. (Ess.)
  • Support for denying a ‘where are you/esp?’ capability from another ECID. (Ess.)

Storing of capabilities

  • Support to store a local list of buddies that have granted you a capability. (Ess.)
  • Support to store a local list of buddies that you have granted a capability. (Ess.)

 

@esp

  • Ability to issue an @esp to a buddy that has already granted the appropriate capability. (Ess.)
  • Ability to do the above irrespective of requesters or requestees location within the Palace. (Ess.)
  • Inability to to issue a successful @esp to an ECID that has not granted that capability. (Ess.)
  • Ability to perform multiple @esps to the same ECID in a contiguous (on-line?) Palace seesion without invoking multiple calls to PLS [implies local caching]. (NTH.)

@whereis

Note: these will be implicit in an @esp; the priority here refers to an explicit command available in The Palace.
  • Ability to issue an @whereis to find a buddy that has already granted the appropriate capability. (NTH.)
  • A successful @whereis returns the Palace:room location of the buddy. (NTH.)
  • Inability to to issue a successful @whereis to an ECID that has not granted that capability. (NTH.)
  • Inability to to issue a successful @whereis to an ECID that has granted that capability and subsequently revoked it. (NTH.)

Note: we need response times to make these meaningful; it would also be useful to define ‘heavyweight’ and ‘lightweight’ in absolute units of some sort.

Load testing

PLS

The following is based on these assumed ICQ numbers:

- 12,000,000 registered ICQ UINs,

- 400,000 are online at any one time (3.333*%),

- 2 million new sessions per day (16.7%).

For the PLS we will assume higher usage per ECID to give us a design margin add more variables. Also just used # online to keep it simple.

- 1,000,000 ECIDs,

- 10% active at any one time = 100,000,

- average session length = 1 hour (conveniently).

- 0.1% of active users will request a capability per hour = 10,000.

- Each user will request esp/whereis once per session hour.

- 25% start sessions per day = 250,000,

So,

  • Number of registrations per hour:
    • ((# active / average session lengh hrs) * factor for re-registrations)
    • e.g. (100,000 / 1) * 1.5 = 150,000.
  • Number of capability requests per hour
    • # active * % requesting a capability
    • 100,000 * 0.1 = 10,000
  • Number of esp/whereis per hour
    • # active * # number of requests made per session hour
    • 100,000 * 1 = 100,000
  • Unregistering?

 

Configuration testing

Minimum functionality will be verified in each configuration.

Operational testing

Combining system testing with operational testing would give us the best return in this compressed timeframe; this means that testing the full system (Palace, ECID, PLS) would take place in the actual operational environment.

 

 

Release acceptance criteria

Functionality and attributes

The functions and attributes in ‘Features to be tested’ must be categorized into:

  • Essential,
  • Nice to have (non-essential).

Essential items: these must be present in the product and meet the minumum stability criteria defined below. If these standards are not met the product cannot be released.

Note: for load criteria we need some maximum acceptable response times for each function.

Nice-to-have items: these must be present in the product and meet the minumum stability criteria defined below to be considered part of the release. If they do not meet the criteria then they are not considered part of the release.
