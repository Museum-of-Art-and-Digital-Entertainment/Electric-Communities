Telepathy Manager

Introduction

This subsystem manages the addressing, sending, and displaying of telepathy messages.

Related Documents

Requirements

Support for unums and ingredients, identity, UI (balloons), and address book.

Architecture

Current Architecture Overview

The ingredient works with the UI to manage the entering and addressing of telepathy balloons, including the completion of nicknames.

The ingredient also requests and grants (and refuses) telepathy capabilities (identifacets).

And finally, the ingredient uses identifacet to send, and ultimately display, messages.

Proposed Architecture Overview

The code includes some (but not complete) support for Groups. Address Book does not currently support groups.

Currently, a new telepathy capability is obtained for every message, even if we are already holding a capability from the recipient. Reusing capabilities could make subsequent messages go faster at the cost of making error handling more complex.

More care can be taken in development of the KnownAvatarList to favor the most recently interacted with in name completion.

Remove EExceptions and ewhens in favor of explicit messages.

The ingredient contains strings which should be internationalized.
