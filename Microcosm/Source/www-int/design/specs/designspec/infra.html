Infrastructure Services

In considering the products we're going to ship, it became clear to me that there is more involved than is clearly defined as the end user system (and tools) MicroNet, and the Hub product (and tools.) In considering What Components We Need In Order To Ship MicroNets and Hubs, we need to name the components in a third group: Infrastructure Services.

Infrastructure services are the programs that are used by just about every kind of host; single machine local turfs, small hubs, or major commercial service farmsÖ The internet already provides infrastructure services such as DNS (Domain Name Service,) sendmail, etc. Like with our Infrastructure Services, internet services are not always tightly bundled with/offered by every network node. So, initially considering these services as separate from a Hub or MicroNet product bundle, imagining them running `somewhere in the net' is helpful for defining their modularity and scope of complexity. Then we can consider, separately, which Infrastructure services we initially supply with first product and which we choose to host at EC.

(Note: These notes override the previous Hub Tools notes, which in some places confused tools with some of these services.)

The EC Infrastructure Service Modules:

PLS (Process Location Service)

Maps object identifier fingerprints to IP addresses.
Object Title Service (Four flavors: Host, Owner, Scarcity, Banks)

Trusted 3rd parties for transactions.
Object Directory Service (avatars, telepads, profiles, etc.)

Maps common names to object identifiers, resolved by PLS.
Resource Location Services (Images, Models, Sounds, etc.)

Where to get resources, in case you can't find them locally.
Facet Transfer Services (PO Boxes)

Where to pick up facets granted when you weren't connected.
Certification Services (Issuance/Verification)

Some 3rd party providers already exist. (Verisign.)

---

Deployment scenarios:

#1 EC initially provides ALL infrastructure services

PRO: Simplifies Hub and MicroNet installation (no IS shipped.)
Modules can still have yucky interfaces.
Modules can run on single (Sun) platform exclusively.
CON: Limits hub capabilities to EC policies/support load.
EC provides these services to everyone, potentially a huge load
Central services hosting subject to single point of failure.
(EC goes down, everyone goes down.)
Central services hosting is a human-resource (support) load.

#2 EC provides minimal infrastructure services

PRO: Removes EC from dependency chain with Hub providers.
Limits EC operational overhead (hardware and personnel)
Gets us to ISP realm hosting sooner.
CON: Modules must be multi-platform, stable & easy to use.
Complicates installation.
Complicates Hub operation significantly (support.)

#3 Hybrid: EC hosts ALL services, but provides a subset in specific

products.

#3a: Hub includes: PLS, Directory, and Resource Location Services.

MicroNet includes: Local Directory (In all packages, always.)

PRO: Provides for `Palace' level hub hosting.
CON: Members still use EC for Title, Facets, & certificates

#3b: Hub includes: ALL Services

MicroNet includes: PLS, Directory, Resource Location Services

PRO: Hub-less MicroNets behind fire walls possible
Hub is full-fledged service and infrastructure provider
Full ISP hosting
CON: See scenario #2

#3c: #3b minus Currency & Scarcity Title Services [Certificate issuing?]

PRO: Less complicated. Few care about tokens and scarcity.
