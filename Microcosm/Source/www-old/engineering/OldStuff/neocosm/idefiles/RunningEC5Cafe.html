Running EC5 in Visual Cafe

Last updated 1998/06/30..

This is just a start at real documentation. I (wsf) cribbed most of this from Running EMain in Visual Cafe.

NOTE: There were some differences between 2.0 and 2.5, but only in the format of windows and subwindows. I have included instructions for both versions.

  1. Get the CD or use Scott's download and install it.
  2. Edit the sc.ini file to add our .dlls to the PATH. See Tony and John's notes for details.
  3. Do a "cvs up -dP" in /ec5/src/javasrc.
  4. Do a "cvs up -dP" in /ec5/projects.
  5. Start up the Symantec Cafe IDE and use 'Open Project' on the 'File' menu to open  '/ec5/projects/ec/neocosm.vep'
  6. Select the 'Project - neocosm.VEP' window and select 'Build Application'
  7. Select the 'Execute' menu item from the Project menu
NOTE: I could not get EMain to run in the debugger. It appears as if it does not open the little black Java window fast enough before it also gets to java.io.FileInputStream#read(byte, int, int) which then calls the native method readBytes(byte, int, int) and hangs.

Various evils in the environment and work arounds (if any)

  • It appears that if your main thread finishes before the threads you are testing, it hoses the debugger big time. The work around seems to be to set up a "wait forever" at the end of your test driver. i.e. synchronize on an object and then call it's wait method. You will have to catch the InterruptedException as well.
  • If the compiler complains it can't open one of your .java files, open it manually and make a trivial change (e.g. insert and delete a blank.) It will then compile. You may have to do this procedure for every make.
  • If you get a "can not write" for a .class file, check to see if NT has a lock on it. I (wsf) got mine after some hard debugger crashes. The fix is to reboot NT.
  • Stepping over a method invocation which throws an exception does not stop. Have fun watching your exception handling code run at full speed.
