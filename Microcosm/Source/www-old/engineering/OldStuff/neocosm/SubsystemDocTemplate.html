This template is an aid for writing our subsystem inventory documentation. The first version of these documents are intended to capture what we have today and it is very important that we keep them updated. The implementation details may not be filled in but the design details should be checked in. It's OK not to know everything to check the documentation as "what I know now, about this subheading", as well as indicating areas that still need documenting.

These documents are stored on the internal web site as a permanent record of our design decisions. The parts of the system that move forward will go through the design review process started by Trev.

If the subsystem in question already has documentation please see if you can fit it in this format, as Trev reminded us having a standard format makes the documents easier to read for everyone while avoiding having to make a number of trivial decisions.

Remember that documents are written once and read many times. Writing a thorough, concise document is like writing good code - it will save you and the people who use it a lot of time later on (and will make the design inspection process much easier). Consult with Lani and Amy for help with writing. Also, see their Documentation Standards pages.

Everything below this paragraph, apart from the final link back to the index, is part of the template. All the text in italics is to be replaced by your (non italicized) text.

Revisions to the document should be reflected in the last updated field on top of the document and tagged comment of the form [yy/mm/dd reviser]. (suggestions for other amendment marking notation welcome)

Introduction

The introduction should give a brief very high level overview of what the design is for and how it fits with other components. It should be possible for someone to read just the introduction and get a good sense of whether they need to read the rest of the document. The introduction sets the scene, but doesn't fill it in at all

Related Documents

(Optional) list any related design documents

Requirements

Listing the subsystem's initial requirements, also please add notes on requirements that are met by the current version of the system, as well as known not-met potential requirements, e.g. firewall support, etc.

Architecture

Diagrams are strongly encouraged; a few diagrams can do wonders for clarifying an architecture. If you don't know how to add diagrams consult Lani and Amy.

Current Architecture Overview

The introduction set the scene; this section gives the key properties of the current design. It may mention classes and some important interfaces but it doesn't go into details. Someone should be able to read just the overviews of our design documents and get a good understanding of how the system works and fits together, without knowing the full details. Also include any insights gained from experience like design bugs or expense discovered in usage.

Proposed Architecture Overview

This section is not urgent for the first version of this document, you should capture what direction you feel this subsystem ought to take. As we start addressing specific systems this section will grow into the core of the design document.

Off the shelf alternatives

Any notes about potential off the shelf technologies that could be used, and their trade-offs.

Other Design Objectives, Constraints and Assumptions

Lists any special objectives and assumptions of the code e.g. reusability, thread safety, security, performance, use of resources, compatibility with existing code etc. This section gives important context for reviewers

Current implementation

This section should give details of the major classes and interfaces.

Which packages on our tree does this subsystem cover?

Implementation overview

In many cases, this section can link to JavaDoc output from actual Java classes and interfaces. This saves writing documentation twice (the designers will have to JavaDoc their interfaces anyway). The JavaDoc should be linked into the design document. Chip's JavaDoc style guidelines explain how to use JavaDoc effectively.

Exception handling

For any clients of the subsystem this section discusses in general the exceptions that this subsystem can throw in usage, this section should also link to the corresponding JavaDoc.

Examples

Are there examples?

Performance

Lists performance issues, targets, considerations and tradeoffs for this subsystem. These issues can include: class bloat, runtime resource consumption and usage/invocation cost. Here are some questions to consider:
  • Is this code doing the minimum possible for TheProduct?
  • Could this code be native?
  • Should this code be native?
  • What do you see as the most likely performance bottlenecks in this code?
  • Is there another way to write them so they're not bottlenecks?

Testing Strategy and Debugging

Describes the testing strategy for this subsystem. List if it is possible to stress test this code outside TheProduct. Lists the testing API, any tests and debugging utilities which are to be developed to help test the design (e.g. test classes, trace categories, etc.).

Resolved Issues

Open Issues

This section of the document is used by the authors and moderator to store any incomplete information - issues identified during a design inspection but not yet resolved (the task list), notes that aren't ready to be put into the main text, etc.
