I have some comments on the DObjects testing doc (now that I've had a

chance to read it).

 

In general, it looks good to me. I agree with the "Level of Effort"

assessments, and scenarios being the highest priority testing task.

 

For Scenarios:

 

What's there seems fine to me, but I would suggest an addition: that some

of the scenarios be constructed from simple, but high-level usage scenarios

(e.g. one, two, three or few DObjects, hosted on one or a couple of the

participating machines, with repeated changes of session membership, for

example). Note I'm just suggesting this as an addition to the scenario

generation that's already there.

 

For Network Partition Tests:

 

This looks good. It really is important that these tests be done

thoroughly...we all agree with that. It is hard to decide when to simulate

arbitrary failure (as it could happen at any time, and the system should be

robust under such failures at all times), but I think Brian has articulated

several of the most important partition test cases.

 

For Stress/load/timing tests:

 

I would say the priority of this kind of testing is "Moderately High". I

would hope we could do some of this testing soon, since scalability is

critical, and it's a difficult property to achieve (and it would be good to

identify problems with scalability, if any, as soon as possible).

 

I agree with Brian's tests for this. They definately would be good stress

tests.

 

For Performance:

 

I feel the same way about performance as I do about stress tests...it would

be very useful to do some of them (and I've started doing a little of that

myself yesterday). As much as can be done would be valuable.

 

In Test Support Needed:

 

You say: "TBD: to what extent will the DObjects code be tested with the

comm system stubbed out, and to what extent using the real comm system? My

leaning is toward using the real comm system, except that it will make

class bloat harder to measure."

 

Currently the DObjects code does not use our comm system...primarily

because we don't have a working comm system yet in any form. The DObjects

code, however, does have its own relatively simple comm system, and this

currently works just fine (in my tests) for supporting the functionality of

the actual DObjects subsystem code. The same is true for the asynchronous

messaging infrastructure (i.e. the DObjects code has its own, and that is

working quite well, and is not yet based upon the elib package). Like I

said, both the comm system and the asynch messaging support system that

DObjects uses are working *quite well* (i.e. I've found no bugs in my tests

to this point). I've already made accomodations in the DObjects code for

using the full comm system and the asynch messaging infrastructures, and

moving over to these subsystems should be a small effort, given that they

work reliably, and meet their specific requirements (WRT functionality,

performance, reliability, etc).

 

In any event, I suggest that testing go ahead right now with the DObjects

code as is. When the comm system and asynch messaging structures are

ready, I will move the code over to using these systems, and given that

they fulfill their contracts for usage by the DObjects code, there should

be little to no observable effect to the code that depends specifically

upon the DObjects subsystem functionality (i.e. the subsystems that are

clients of the DObjects subsystem). If there is an effect, then it would

indicate some problem with the comm system, or with the asynch messaging

subsystem, or with their interaction with the DObjects code, and so this

would actually serve as a good test of these systems (along with the

independent testing of those two subsystems, of course).

 

That's all my comments. Like I said, I think it looks good. Let's do it.

 

Scott
